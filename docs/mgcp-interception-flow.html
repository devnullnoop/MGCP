MGCP Interception Flow

Every arrow is a point where MGCP gets to shape the LLM's next action. The goal of this document is to make every interception visible so gaps and weak points stand out. Source of truth for routing: ~/.mgcp/intent_config.json.

1. Big picture — five hook interception points

Claude Code fires five hook events during a session. MGCP owns one Python script per event. Four are advisory (inject text the LLM may skim or ignore); one is enforcing (can refuse a tool call outright).

The advisory-vs-enforcing distinction: SessionStart / UserPromptSubmit / PostToolUse / PreCompact inject text as <system-reminder> tags — the LLM can skim or ignore them (which is how query-before-git-operations failed v1→v4). PreToolUse is different: it returns permissionDecision: deny and the tool call is refused by the harness. Interception = compliance. v2.4 generalizes this: the hook is now a data-driven evaluator reading ~/.mgcp/enforcement_rules.json. Rules are CRUD-ed from chat via six MCP tools (add_enforcement_rule, update_enforcement_rule, …) — no code changes, no release. Seeded default rule: git commit / git push requires query_lessons in the same turn. Scoped bypass via MGCP_BYPASS:<scope> (bare MGCP_BYPASS disables every rule).

2. SessionStart — session-init.py

Single shot per session. As of v2.5, injects only the session-start bootstrap checklist and workflow execution discipline — the routing/actions contract is owned by user-prompt-dispatcher.py, which re-injects it on every message (see section 3). v2.5 dropped the SessionStart copy because the two injections were identical and burned ~300 tokens per session.

4. PreToolUse — pre-tool-dispatcher.py (ENFORCING · data-driven)

Fires before every tool call. The only MGCP hook that can actually refuse a tool call — all others inject text the LLM can skim past. Returns permissionDecision: "deny" with a reason the LLM sees on its next turn.

v2.4: no hardcoded rules. The hook loads every rule from ~/.mgcp/enforcement_rules.json and applies each one whose trigger matches and whose bypass_scope isn't in turn_bypass_scopes. Matched rules whose preconditions are unsatisfied contribute to the denial message; all failing rules are reported together.

Rule shape

{
  "name": "git-requires-query-lessons",
  "enabled": true,
  "trigger": {
    "tool_name": "Bash",
    "command_match": {"type": "git_subcommand",
                      "subcommands": ["commit", "push"]}
  },
  "preconditions": [
    {"type": "tool_called_this_turn",
     "tool_name": "mcp__mgcp__query_lessons"}
  ],
  "bypass_scope": "git",
  "deny_reason": "git commit/push requires query_lessons first"
}

Trigger command_match.type ∈ git_subcommand / regex / contains. Precondition type ∈ tool_called_this_turn / tool_not_called_this_turn / staged_files_coupling. The coupling type takes couplings: [{"when_staged": [glob], "require_one_of": [glob]}] — use it to enforce doc-parity, test-parity, or changelog discipline on commits.

Why a token walker, not a regex

A naive \bgit\s+(commit|push)\b regex false-positives on grep 'git commit' docs/ or echo "how to git commit". The hook uses shlex.shlex(..., punctuation_chars=True) so quoted strings stay as single tokens and shell operators (;, &&, ||, |, ()) split into their own tokens. The walker only matches git <subcommand> when git appears at a true command boundary.

State flow across the three hooks that collaborate

HookOperation on turn state
UserPromptSubmitReset turn_tools_called = []. Parse every MGCP_BYPASS[:<scope>] token in the prompt into turn_bypass_scopes: list[str].
PostToolUseAppend tool_name to turn_tools_called (any tool, not just query_lessons).
PreToolUseRead both fields. Apply every enabled, triggered, non-bypassed rule.

Fail-open policy

Any parse error (unterminated quotes, malformed rules JSON, unknown precondition type) results in that rule being skipped, never a blanket deny. A completely missing enforcement_rules.json allows all tool calls. The exception: a valid rule whose preconditions aren't met denies — that's the whole point.

7. The growth loop — how interception learns over time

Each interception can produce data that shapes the next interception. This is the core value prop: MGCP gets better the more it's used.

add_catalogue_item
save_project_context --> DB[(SQLite
+ Qdrant)] DB -- embeddings --> QL[query_lessons] QL -- surfaces at task start --> LLM DB -- graph relationships --> REM[REM cycle] REM -- detect_communities --> IC[intent_calibration] IC -- proposed_patch --> CFG[intent_config.json] CFG -- read by --> SI[session-init.py] CFG -- read by --> UP SI -- injects routing --> LLM style LLM fill:#1c2a3e,stroke:#7ab8ff style DB fill:#171b22,stroke:#2a3140 style CFG fill:#2e281b,stroke:#ffb86b style REM fill:#1b2e1b,stroke:#88cc88 style PT fill:#3a1f1f,stroke:#ff7777

Three feedback paths close the loop:

  1. Session → lessons → next session. Mistakes become add_lesson calls, surface via query_lessons in future sessions.
  2. Lessons → REM → routing config. Community detection finds tag clusters that span multiple intents. intent_calibration proposes config patches.
  3. Routing config → hooks → session start. Adding an intent in JSON changes hook injection immediately next session — no code commit required.

9. Known gaps (candidate improvement areas)

#StatusGapSymptomPossible fix
1 open Keyword-only gating Synonyms/paraphrases slip through ("ship it", "lock this down", "tuck it in") Semantic intent matching via BGE; daemon-hosted model to avoid per-prompt cold start
2 partial ✓ No enforcement, only advice Lesson query-before-git-operations failed v1→v4 across months Done for git commit/push via PreToolUse gate. Extend to Edit/Write → add_catalogue_item, and to doc-parity (gap #9).
3 open Only git + session_end have keyword gates Catalogue-security, catalogue-decision, task_start all rely on LLM self-routing Add keyword patterns to more intents in intent_config.json
4 open First-message bootstrap fragility "Tough" first questions cause skipping of get_project_context / query_lessons Track whether context-load tools have been called this session; nag harder in dispatcher if not
5 mechanism ✓ PostToolUse is best-effort Edit/Write checkpoint emitted every time but often skipped by LLM v2.4 enables this: one add_enforcement_rule call with trigger: Edit and precondition: tool_called_this_turn(add_catalogue_item). Not enabled by default because it's aggressive; users opt in per-project.
6 open Routing prompt gets long 8 intents × description + action = ~600 tokens per UserPromptSubmit re-inject Adaptive injection: only emit intents whose tags match the prompt, fall back to full list when ambiguous
7 resolved ✓ Windows hook portability settings.json hardcodes python3; Windows default is python Done in v2.4. _hook_python_command() returns shlex.quote(sys.executable), so installed hooks use the same interpreter that ran mgcp-init — portable across Windows / venvs / paths with spaces.
8 open No telemetry on gate effectiveness Can't tell which gates are working vs being ignored Log gate-fire + deny events to telemetry.db alongside whether the required tool was actually called next turn
9 mechanism ✓ Doc/code parity not enforced Code changes land without updating architecture docs that describe them v2.4 ships the staged_files_coupling precondition for exactly this. One add_enforcement_rule call per coupling, e.g. "if src/mgcp/hook_templates/* is staged, require docs/mgcp-interception-flow.html". Not shipped as a default because couplings are project-specific; maintainers add the rules matching their repo layout.
10 new Bypass token is all-or-nothing MGCP_BYPASS disables every PreToolUse gate for the turn — resolved in v2.4 Done. Each rule has a bypass_scope; MGCP_BYPASS:<scope> disables only matching rules; bare MGCP_BYPASS still disables all for convenience.

10. Recipes — the remaining open gaps are one-liners

v2.4 turned gaps #5 and #9 into chat-time config changes. Below are the exact calls a maintainer can make to close them for their project. These are examples — pick the ones that fit your workflow.

Doc/code parity (gap #9)

add_enforcement_rule(
  name="hooks-require-flowchart-update",
  trigger={"tool_name": "Bash",
           "command_match": {"type": "git_subcommand",
                             "subcommands": ["commit"]}},
  preconditions=[{
    "type": "staged_files_coupling",
    "couplings": [{
      "when_staged": ["src/mgcp/hook_templates/*"],
      "require_one_of": ["docs/mgcp-interception-flow.html"]
    }]
  }],
  bypass_scope="docs",
  deny_reason="Hook template changed; stage the interception flowchart too "
              "or run MGCP_BYPASS:docs to skip."
)

CHANGELOG discipline

add_enforcement_rule(
  name="src-requires-changelog",
  trigger={"tool_name": "Bash",
           "command_match": {"type": "git_subcommand",
                             "subcommands": ["commit"]}},
  preconditions=[{
    "type": "staged_files_coupling",
    "couplings": [{
      "when_staged": ["src/**/*.py"],
      "require_one_of": ["CHANGELOG.md"]
    }]
  }],
  bypass_scope="changelog",
  deny_reason="Code change without CHANGELOG entry. Bypass with MGCP_BYPASS:changelog."
)

Edit requires catalogue capture (gap #5)

add_enforcement_rule(
  name="edit-requires-catalogue-capture",
  trigger={"tool_name": "Edit"},
  preconditions=[{
    "type": "tool_called_this_turn",
    "tool_name": "mcp__mgcp__add_catalogue_item"
  }],
  bypass_scope="edit",
  deny_reason="Capture the pattern/gotcha/coupling you learned before editing. "
              "Bypass with MGCP_BYPASS:edit for trivial edits."
)

This one's aggressive — it blocks every Edit until you call add_catalogue_item. Consider starting with toggle_enforcement_rule(name=..., enabled=False) and enabling only during deliberate capture sessions.
